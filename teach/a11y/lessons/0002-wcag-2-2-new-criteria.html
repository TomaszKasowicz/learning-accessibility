WCAG 2.2's new criteria — and what axe can’t catch

WCAG 2.2 (Oct 2023) added 9 success criteria and removed one (4.1.1 Parsing). Six of the new ones are Level A/AA — inside your bar. This is the "recent rules" gap you flagged. The twist that matters for your skill: axe can only automate one of them.

Why this matters for the skill: you said the bar is "axe default rules." But axe's defaults exclude WCAG 2.2 rules, and 5 of the 6 new A/AA criteria can't be automated at all. A skill that only runs axe defaults will silently pass code that violates WCAG 2.2. You need to (a) opt in to the 2.2 rule, and (b) add a manual checklist. This lesson gives you both.

1. The six new A/AA criteria

Each card: what it means, how you fix it, and whether axe can detect it automatically.

3.2.6 Consistent HelpA

If a help mechanism (contact link, chat, FAQ) repeats across pages, it must appear in the same relative order every time.

Fix: put help links in a shared layout/component so position is consistent. Angular: a shared header/footer component, not per-page placement.

Manual only — axe can't judge "same order across pages"

3.3.7 Redundant EntryA

Don’t make users re-enter info they already gave earlier in the same process — auto-populate it or let them pick it.

Fix: carry form state across multi-step flows (e.g. "billing same as shipping" checkbox, prefilled fields).

Manual only — needs flow context

2.4.11 Focus Not Obscured (Minimum)AA

When an element receives keyboard focus, it must not be entirely hidden by other content (sticky headers, cookie banners, toasts).

Fix: use scroll-margin, avoid covering focused content with fixed overlays; ensure sticky bars don’t fully hide the focused control.

Manual / DevTools Pro — not in axe-core

2.5.7 Dragging MovementsAA

Anything done by dragging must have a single-pointer alternative (tap/click) — unless dragging is essential.

Fix: add click/keyboard alternatives to drag-to-reorder, sliders, drag-and-drop (e.g. up/down buttons, +/- steppers).

Manual only — intent can't be inferred

2.5.8 Target Size (Minimum)AA

Pointer targets must be at least 24×24 CSS px, or have enough spacing that a 24px circle around each doesn’t overlap a neighbour.

Fix: enlarge small icon buttons / inline links; add padding or spacing. The classic offender: tightly packed icon-only buttons.

✓ axe CAN auto-detect — rule target-size (the only 2.2 rule)

3.3.8 Accessible Authentication (Minimum)AA

Don’t force a cognitive function test (memorise/transcribe a code, solve a puzzle) to log in, unless there’s an alternative.

Fix: allow password managers / paste in fields; support passkeys/OAuth; don’t block autofill on auth inputs.

Manual only

Sources: W3C What’s New in 2.2 · Deque WCAG 2.2 (fixes + code). (The other 3 new criteria — 2.4.12, 2.4.13, 3.3.9 — are AAA, out of scope for now.)

2. The automation reality (this is the lesson)

Deque: "because how few WCAG 2.2 criteria can be automated without false positives, target-size is likely the only WCAG 2.2 rule in axe-core."

Gotcha your skill must handle: axe-core ships WCAG 2.2 rules disabled by default. Running plain axe.run() will not test target-size. And the wcag22aa tag alone won't run it either — a tag only selects among enabled rules; it doesn't flip a disabled rule on. You must enable the rule explicitly:
// The tag SELECTS; rules:{...} ENABLES. You need both. axe.run(document, { runOnly: { type: 'tag', values: ['wcag2a','wcag2aa','wcag21a','wcag21aa','wcag22aa','best-practice'] }, rules: { 'target-size': { enabled: true } } // <-- this is what turns it on });
Engine vs. product — reconciling what you may have seen: the "off by default" rule above is true of the raw axe-core engine (what @axe-core/playwright runs) — empirically verified on axe-core 4.11.x: target-size is the only wcag22aa rule and it ships enabled: false. Some product layers — the axe DevTools browser extension / Pro, and some CLI wrappers — ship their own config and may default WCAG 2.2 on. So if a DevTools UI shows target-size on out of the box, that's the product's default, not the engine's, and it does not carry into your Playwright run. Enabling it explicitly (above) is correct and idempotent either way. Verified: rule-descriptions.md + local axe-core._audit.rules check.
Design takeaway for the skill: "pass axe" ≠ "WCAG 2.2 compliant." Your skill needs two arms: an automated arm (axe with the wcag22aa tag enabled) and a manual checklist arm prompting a human for the 5 non-automatable criteria.
