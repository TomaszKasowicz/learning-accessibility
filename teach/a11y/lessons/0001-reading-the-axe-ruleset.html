Reading an axe violation

Before a skill can fix accessibility bugs, it has to read what the tools report. Every tool in your pipeline (axe DevTools, Lighthouse, pa11y, Playwright+axe) speaks the same dialect underneath: the axe-core result object. Learn to read one, and you can read all of them.

Why this, first? Your mission is a skill that runs these tools and fixes code to pass axe's default ruleset. The default ruleset is just a set of tags, and each violation is a JSON object with an impact and a WCAG mapping. This lesson makes that object legible — the literacy everything else builds on.

1. The default ruleset is a set of tags

When you run axe with no restriction, it executes every rule tagged with one of these. This is your compliance bar:

TagMeans
wcag2a / wcag2aaWCAG 2.0 Level A / AA
wcag21a / wcag21aaWCAG 2.1 Level A / AA
wcag22aaWCAG 2.2 Level AA (the recent stuff you flagged)
best-practiceNot required by WCAG, but widely agreed good practice

A rule also carries an SC tag like wcag111, which decodes as success criterion 1.1.1. So wcag143 → SC 1.4.3 (Contrast). That's your tag→WCAG decoder ring.

Source: axe-core API.md — tags table. To restrict to this bar explicitly: runOnly: { type: 'tag', values: ['wcag2a','wcag2aa','wcag21a','wcag21aa','wcag22aa','best-practice'] }.

2. Anatomy of one violation

Here's a single entry from results.violations. The annotations call out the five fields you'll read every single time:

"id": "button-name", // ① the rule — what failed "impact": "critical", // ② severity → triage order "tags": ["cat.name-role-value", "wcag2a", "wcag412"], // ③ standard + SC 4.1.2 (Name, Role, Value) "help": "Buttons must have discernible text", "helpUrl": "https://dequeuniversity.com/rules/axe/4.10/button-name", // ④ how to fix it (always present) "nodes": [{ "target": ["button.icon-btn"], // ⑤ which element (CSS selector) "html": "<button class='icon-btn'><svg.../></button>", "failureSummary": "Fix any of: element has no text; no aria-label; no aria-labelledby..." }]

Read it as a sentence: "Rule button-name (critical, WCAG 2.0-A, SC 4.1.2) failed on button.icon-btn — it has no accessible name. Fix per the helpUrl." That's the whole skill loop in miniature: identify → locate → fix → re-run.

3. Impact = your fix order

ImpactTriage
criticalBlocks a user entirely. Fix first.
seriousMajor barrier. Fix next.
moderateFrustrating but workable.
minorPolish.
Don't forget the other buckets. A run returns four groups: violations (fix), incomplete (axe is unsure — a human must check), passes, and inapplicable. A good skill reports incomplete too, because that's where automated tools hand off to manual testing.
