Fixing the top axe violations in Angular

A handful of rules account for the bulk of real-world axe failures. Learn their before/after in Angular templates and you can clear most of a component by hand — exactly what your skill will automate.

Why this: your skill's loop is read → locate → fix → re-verify (Lesson 1). This lesson is the fix step, in your primary stack. Each card maps an axe rule → the Angular template change. The ✎ line shows which @angular-eslint rule catches it at lint time — your skill's cheap first pass.
image-alt

Images must have alt text

critical SC 1.1.1 · wcag2a
Before
<img [src]="user.avatar">
After
<img [src]="user.avatar" [alt]="user.name + ' avatar'"> <!-- decorative? empty alt: --> <img src="divider.svg" alt="">

Why: screen readers announce the alt. Informative images need a meaningful name; purely decorative ones need alt="" so they're skipped. Bind dynamic alts with [alt].

✎ lint: @angular-eslint/template/accessibility-alt-text

button-name

Buttons must have discernible text

critical SC 4.1.2 · wcag2a
Before
<button mat-icon-button (click)="close()"> <mat-icon>close</mat-icon> </button>
After
<button mat-icon-button (click)="close()" aria-label="Close dialog"> <mat-icon>close</mat-icon> </button>

Why: an icon glyph has no text content, so the button's accessible name is empty. aria-label supplies it. Icon-only buttons are the #1 source of this violation.

✎ lint: partly caught; rendered-DOM scan (axe) is the reliable catch

label

Form fields must have labels

critical SC 4.1.2 / 1.3.1 · wcag2a
Before
<input type="email" [(ngModel)]="email" placeholder="Email">
After
<label for="email">Email</label> <input id="email" type="email" [(ngModel)]="email">

Why: placeholder is not a label — it vanishes on input and isn't a reliable accessible name. Associate a real <label for> ↔ id. (Angular Material: use <mat-form-field> with <mat-label>.)

✎ lint: @angular-eslint/template/accessibility-label-has-associated-control

link-name

Links must have discernible text

serious SC 4.1.2 / 2.4.4 · wcag2a
Before
<a [routerLink]="['/post', id]"> <mat-icon>arrow_forward</mat-icon> </a>
After
<a [routerLink]="['/post', id]" [attr.aria-label]="'Read ' + title"> <mat-icon>arrow_forward</mat-icon> </a>

Why: same as buttons — icon-only links have no name. Note [attr.aria-label] for a bound value (plain aria-label for a static string). Avoid "click here" text too (2.4.4).

color-contrast

Text must meet contrast minimums

serious SC 1.4.3 · wcag2aa
Before
/* #9aa0a6 on #fff = 2.6:1 */ .hint { color: #9aa0a6; }
After
/* #5f6368 on #fff = 5.7:1 */ .hint { color: #5f6368; }

Why: needs ≥ 4.5:1 for normal text, 3:1 for large (≥24px, or ≥18.66px bold). This is computed from rendered styles, so only a DOM scan catches it — lint can't. Fix in the design tokens, not per-component.

aria-valid-attr-value / aria-required-attr

ARIA must be valid & complete

serious SC 4.1.2 · wcag2a
Before
<div role="checkbox" (click)="toggle()"> Subscribe </div>
After
<div role="checkbox" tabindex="0" [attr.aria-checked]="subscribed" (click)="toggle()" (keydown.space)="toggle()"> Subscribe </div>

Why: a declared role brings required states & keyboard behaviour. role="checkbox" needs aria-checked, focusability (tabindex="0"), and key handling. Better still: use a native <input type="checkbox"> and skip all of it.

✎ lint: @angular-eslint/template/accessibility-role-has-required-aria, ...valid-aria

Pattern across all six: most fixes are either "give it an accessible name" or "use the native element instead of a div+ARIA." When in doubt, reach for native HTML — it ships roles, states, and keyboard behaviour for free.
