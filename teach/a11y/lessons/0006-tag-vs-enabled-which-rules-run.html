Tag vs enabled: which rules actually run?

You commented out rules: { 'target-size': { enabled: true } } in axe.config.ts but kept the wcag22aa tag — and you still get a serious target-size violation on /pointer-size. That's not a bug. It exposes how axe really decides what to run, and it overturns a note we'd written down. Let's nail the actual model.

Why this matters for the mission. Your skill must honestly opt into WCAG 2.2. If you believe "the tag won't run target-size unless I also enable it," you'll mis-explain the skill and mis-debug scans that do fire the rule. Get the selection model right and you can predict any config's behavior without running it.

Builder API = same logic. Row F uses @axe-core/playwright's .withTags([…]) instead of raw .options({ runOnly }) — and behaves identically, because .withTags() is just sugar for runOnly: { type: 'tag' } (and .withRules() for rules). One gotcha: .options() replaces withTags/withRules, so don't mix them — pick one style.

2. The model that explains every row

axe has two independent switches for "does this rule run?", and a rule runs if either turns it on:

Switch 1 — the tag, via runOnly. When you pass runOnly: { type: 'tag', values: [...] }, axe runs every rule whose tags intersect your list. Selecting by tag is an explicit opt-in; the rule's own enabled flag is not consulted on this path. target-size is tagged wcag22aa, so naming that tag runs it. → explains A, B, E, and why C doesn't.

Switch 2 — the enabled flag, on the default path. When you pass no runOnly, axe runs every rule whose metadata is enabled: true (and isn't experimental). target-size ships enabled: false, so the bare default skips it. → explains D.

// target-size's metadata — confirmed in YOUR node_modules/axe-core@4.12.0
require('axe-core')._audit.rules
  .find(r => r.id === 'target-size')
  // .enabled  => false        ← only matters on the no-runOnly path
  // .tags     => ['cat.sensory-and-visual-cues', 'wcag22aa', 'wcag258']
Myth (what record 0001/0002 said): "target-size is enabled:false, so the wcag22aa tag alone won't run it — you must add rules:{ 'target-size': { enabled:true } }."
Truth: The wcag22aa tag in runOnly runs it on its own. The enabled:false flag only suppresses it on the no-runOnly default run. The explicit rules:{} enable is redundant when the tag is present — useful only as defence for scans that don't request the tag (row E).

3. So should you keep the explicit enable?

Yes — but know what it buys you. It's idempotent and harmless. Its real value is row E: a teammate who scans with wcag2a…wcag21aa but forgets wcag22aa still gets target-size. It guards the bar against an incomplete tag list — it is not "the thing that turns the rule on" when you already pass wcag22aa.

One claim this does not change. Passing axe with target-size on still isn't "WCAG 2.2 AA compliant." Four of the new 2.2 A/AA criteria (2.4.11, 3.2.6, 3.3.7, 3.3.8 — and 2.5.7) aren't automatable. The skill's manual checklist arm still stands. See reference/glossary.html and the manual checklist.
