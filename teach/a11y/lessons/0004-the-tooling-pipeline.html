The a11y tooling pipeline

No single tool finds every bug. Automated tools catch only ~30–50% of WCAG issues. The skill works because it layers tools so each catches what the cheaper one missed — and hands the rest to a human.

Why this: your skill orchestrates several tools. To orchestrate them well you must know each one's job and blind spot. This lesson is the map: three layers, cheapest-and-earliest first, each a wider net than the last.

1. The three layers (a funnel)

Run them in this order. Each layer is slower but catches a class of bug the previous one is blind to.

① Static analysis (lint)

editor + CI · sub-second
Tools: @angular-eslint/template/accessibility-* (Angular), eslint-plugin-jsx-a11y (React)
Reads source/JSX/templates without rendering. Zero runtime cost, instant editor feedback.
Catches: missing alt, label-less controls, invalid ARIA attrs/roles, positive tabindex, click-without-key handlers.
Misses: anything computed at runtime — contrast, dynamic ARIA state, focus order, whether a name is meaningful.
▼

② Rendered-DOM scan (axe engine)

CI / test run · seconds
Tools: @axe-core/playwright, pa11y / pa11y-ci, axe DevTools, Lighthouse (a subset of axe)
Runs axe-core against the real accessibility tree after render. This is where your "pass axe default rules" bar is enforced. (Remember L2: add the wcag22aa tag.)
Catches: computed color contrast, missing accessible names on rendered components, ARIA state mismatches, duplicate ids, landmark/heading structure.
Misses: is the alt text correct? is focus order logical? does the keyboard flow make sense? does it sound right in a screen reader?
▼

③ Manual verification (human)

before merge · minutes
Tools: keyboard only (Tab/Shift-Tab/Enter/Space/Esc/arrows), screen reader (VoiceOver on macOS, NVDA on Windows)
The only layer that judges meaning and experience. Required for the 5 non-automatable WCAG 2.2 criteria (L2) and for confirming a fix actually helps.
Catches: meaningless/wrong names, illogical focus order, keyboard traps, drag-only interactions, focus hidden behind sticky bars, "is this usable?"
Misses: nothing automatable — but it's slow, so let layers ① and ② clear the cheap stuff first.

Two clarifications people get wrong:
• Lighthouse is not a separate engine — its a11y audit runs a subset of axe-core rules and weights them into a score. Use it for a quick page signal, not as your gate.
• pa11y can run two engines (axe-core and HTML CodeSniffer) side by side — useful for CI cross-checking, but it still won't replace the manual layer.

3. Where this lands in your skill

Your WCAG-fixing skill mirrors the funnel: run lint → auto-fix what's safe; run axe (with 2.2 tag) → fix violations using L3 patterns; then emit a manual checklist for the human (the items only layer ③ can confirm). It should never claim "compliant" from layers ① + ② alone.
