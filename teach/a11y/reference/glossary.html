A11y & axe Glossary

The shared vocabulary for this workspace. Every lesson uses these terms exactly as defined here.

Core accessibility terms

WCAG (Web Content Accessibility Guidelines)
The W3C standard for web accessibility. Organised as POUR principles → guidelines → testable success criteria. Current version: 2.2 (Oct 2023). w3.org/TR/WCAG22
POUR
The four WCAG principles: Perceivable, Operable, Understandable, Robust. Every success criterion lives under one.
Success Criterion (SC)
A single testable requirement, numbered principle.guideline.criterion — e.g. 1.1.1 Non-text Content. Each has a conformance level (A / AA / AAA).
Conformance level (A / AA / AAA)
A = minimum, AA = the legal/industry baseline (ADA, EAA, AODA), AAA = enhanced. Our bar maps to A + AA.
Accessibility tree
The browser's parallel-to-the-DOM structure that exposes role, name, state, and value to assistive tech. Most a11y bugs are really "the a11y tree is wrong."
Role · Name · State · Value
What a screen reader announces for an element. Role = what it is (button, checkbox), Name = its label, State = checked/expanded/disabled, Value = current value.
Accessible name
The single string a screen reader announces as an element's label. Computed by precedence: aria-labelledby → aria-label → native (<label>, alt) → text content → title. AccName 1.2
ARIA (WAI-ARIA)
Attributes (role, aria-*) that patch the a11y tree when native HTML can't express semantics. Rule of thumb: prefer native HTML; reach for ARIA only when you must. ARIA APG
Landmark
A region role (banner, nav, main, contentinfo) — usually from native elements (<header>, <nav>, <main>, <footer>) — that lets users jump around a page.

axe-core terms

axe-core
Deque's open-source accessibility rules engine. It powers axe DevTools, Lighthouse's a11y audits, pa11y (optionally), @axe-core/playwright, and more. The engine the skill will target. github.com/dequelabs/axe-core
Rule
One automated check with an id (e.g. image-alt, color-contrast, button-name). A rule runs one or more checks against matching nodes.
Tag
Metadata on a rule. Includes a standard tag (wcag2a, wcag2aa, wcag21aa, wcag22aa, or best-practice), an SC tag (wcag111 = SC 1.1.1), and category tags (cat.forms). You select rules to run via tags. axe API.md
Default ruleset (our bar)
What axe runs when you don't restrict tags: wcag2a, wcag2aa, wcag21a, wcag21aa, wcag22aa, and best-practice. (Excludes AAA, experimental, and section508-only.)
Impact
Severity of a violation. One of: critical serious moderate minor. Use it to triage fix order.
Result groups
axe sorts every node into one of four buckets:
GroupMeaning
violationsFailed a check — fix these.
incompleteaxe couldn't decide — needs a human to check.
passesPassed the check.
inapplicableNo matching elements on the page.
Node / target
The specific element a result applies to. target is a CSS-selector array pointing at it; html is its outer HTML snippet; failureSummary says what to fix.
helpUrl
Per-violation link to Deque University's fix guidance (dequeuniversity.com/rules/axe/<ver>/<rule-id>). Always in the result — your first stop for "how do I fix this?"
check (any / all / none)
Sub-tests inside a rule. A rule passes per its logic: any = at least one must pass, all = every one must pass, none = none may pass.

Tooling layers

Static analysis (lint)
Catches issues in source before render. eslint-plugin-jsx-a11y (React/JSX), @angular-eslint/template/accessibility-* (Angular). Cheapest, fastest, but blind to runtime DOM.
Runtime / rendered-DOM scan
Runs axe against the real a11y tree after render. @axe-core/playwright, axe DevTools, Lighthouse. Catches what lint can't (computed contrast, dynamic ARIA).
Manual verification
Keyboard-only navigation + screen reader (VoiceOver/NVDA). Required because automated tools catch only ~30–50% of issues — they can't judge whether a name is meaningful.
Mental model: Tools check the accessibility tree, not your DOM directly. When a fix doesn't clear a violation, ask "what does this look like in the a11y tree?" — not "what does my markup say?"
